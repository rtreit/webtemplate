Overview

This starter kit separates authentication (proving who you are) from authorization (deciding what you can see). Authentication is delegated entirely to Azure Static Web Apps EasyAuth and Microsoft Entra ID. Authorization is enforced server-side in Azure Functions by inspecting the trusted identity header that EasyAuth injects after sign-in.

The site renders through a client-side gate that hides all content until both checks pass. Unauthorized users see a 403 state. Unauthenticated users see a login prompt.

Authentication flow

Azure Static Web Apps provides a preconfigured Microsoft Entra identity provider. No app registration is required for the default flow. Any Microsoft account can authenticate.

sequenceDiagram participant Browser participant SWA as Static Web App participant Entra as Microsoft Entra ID Browser->>SWA: GET /.auth/login/aad SWA->>Entra: Redirect to Entra login Entra->>Browser: Show credential prompt Browser->>Entra: Submit credentials Entra->>SWA: Return auth token SWA->>SWA: Validate token, create session cookie SWA->>Browser: Set session cookie, redirect to post_login_redirect_uri

Key points

  • The login endpoint is /.auth/login/aad. The post_login_redirect_uri parameter controls where the user lands after login.
  • Adding prompt=login forces Entra to show the credential entry screen even if the user has an active Entra session.
  • The logout endpoint is /.auth/logout. This clears the SWA session cookie but does not clear the Entra browser session. To fully clear both, the site chains logout through the Entra logout endpoint first.
  • The /.auth/me endpoint returns the current user's identity as a JSON payload.

Authorization flow

After authentication, the browser calls /api/check-role to determine whether the signed-in user is authorized to view the site. The Function reads the trusted x-ms-client-principal header, extracts the email domain, and compares it against the configured allowed domain list.

sequenceDiagram participant Browser participant SWA as Static Web App Edge participant API as Azure Function Browser->>SWA: GET /api/check-role (with session cookie) SWA->>SWA: Validate session cookie SWA->>API: Forward request + x-ms-client-principal header API->>API: Decode base64 principal API->>API: Extract userDetails email API->>API: Normalize guest #EXT# identities API->>API: Match email domain against MICROSOFT_EXAMPLE_ALLOWED_DOMAINS API->>Browser: Return { hasSiteAccess, role, email, emailDomain, ... }

Site-wide gate

The base layout (layouts/_default/baseof.html) ships with the real site shell hidden. A JavaScript gate checks auth state and transitions between three visual states:

stateDiagram-v2 [*] --> Loading: Page load Loading --> LoginPrompt: /.auth/me returns no principal Loading --> Forbidden: /api/check-role returns hasSiteAccess=false Loading --> SiteShell: /api/check-role returns hasSiteAccess=true Forbidden --> LoginPrompt: User clicks Switch Account (logout then login) LoginPrompt --> Loading: User completes login SiteShell --> [*]: User browses normally
StateWhat the user seesTrigger
LoadingBrief "Checking your sign-in" panel with Choose Account and Logout linksInitial page load
Login prompt"Sign in to continue" with Login with Microsoft buttonNo authenticated session detected
403 ForbiddenRed 403 panel showing the signed-in email and allowed domain listAuthenticated but email domain not in the allowed list
Site shellFull site with navigation, content, and footerAuthenticated and authorized

A 5-second timeout ensures the gate always resolves to the login prompt if the auth check hangs.

Data flow: x-ms-client-principal

Azure Static Web Apps injects a Base64-encoded JSON header called x-ms-client-principal into every request forwarded to Azure Functions. This header is set by the SWA edge, not by the browser, so it cannot be spoofed by client-side code.

flowchart LR A[Browser] -->|Session cookie| B[SWA Edge] B -->|x-ms-client-principal header| C[Azure Function] C -->|Decode Base64| D[JSON principal] D -->|Read userDetails| E[Email address] E -->|Extract domain| F[Authorization decision]

Principal structure (in API Functions)

{
  "identityProvider": "aad",
  "userId": "abcd12345...",
  "userDetails": "user@microsoft.com",
  "userRoles": ["anonymous", "authenticated"]
}

Important limitation: The claims array is available at /.auth/me (browser-side) but is not included in the x-ms-client-principal header forwarded to API Functions. This is why authorization uses userDetails (email) rather than token claims like tid.

Guest identity normalization

When a user from one Microsoft Entra tenant authenticates against a Static Web App deployed in a different tenant, the userDetails field may arrive in a guest-style format:

randyt_microsoft.com#EXT#@othertenant.onmicrosoft.com

The shared auth helper in api/shared/roles.js detects this pattern and normalizes it back to the original email:

flowchart LR A["randyt_microsoft.com#EXT#@othertenant.onmicrosoft.com"] --> B["Detect #EXT# pattern"] B --> C["Extract local part: randyt"] B --> D["Extract domain: microsoft.com"] C --> E["randyt@microsoft.com"] D --> E

This ensures cross-tenant guest identities are correctly matched against the allowed domain list.

Role system

Beyond the domain-based site gate, the starter kit includes a blob-backed role system for finer-grained access control:

flowchart TD A[User signs in] --> B{Email in roles.json?} B -->|admin list| C[Admin role] B -->|member list| D[Member role] B -->|not found| E[Default: visitor role] C --> F[Can manage roles at /admin/] D --> G[Can see member content] E --> H[Basic site access only]
RoleSourceCapabilities
anonymousBuilt-in SWA roleSee login prompt only
authenticatedBuilt-in SWA rolePass the site-wide gate (if domain is authorized)
visitorDefault app roleSee public project cards
memberAssigned in roles.jsonSee member-only project cards and /members/ content
adminAssigned in roles.json or BOOTSTRAP_ADMIN_EMAILFull access including /admin/ role management

The role configuration lives in Azure Blob Storage at site-config/roles.json. On first run, the system creates a default config with the BOOTSTRAP_ADMIN_EMAIL as the initial admin.

Security boundaries

flowchart TD subgraph "Trust boundary: SWA Edge" A[Token validation] --> B[Session management] B --> C[x-ms-client-principal injection] end subgraph "Trust boundary: Azure Functions" D[Read x-ms-client-principal] --> E[Domain authorization] E --> F[Role resolution] F --> G[Return data or 403] end subgraph "No trust: Browser" H[JavaScript gate] --> I[UI show/hide] end C --> D G --> H
LayerWhat it enforcesTrust level
SWA EdgeToken validation, session cookies, header injectionFully trusted (platform-managed)
Azure FunctionsDomain authorization, role checks, 403 responsesTrusted (server-side, reads platform-injected headers)
Browser JavaScriptUI gating (show/hide site shell)Not trusted (presentation only, can be bypassed)
Static HTMLNothingPublic files, always served to the browser

The real security boundary is the Azure Function. The JavaScript gate and static HTML are convenience layers. Protected data must always be served through authenticated API endpoints, never baked into static HTML.
